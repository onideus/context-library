# Adversarial Review

Adversarial review is the merge gate. Its job is to catch defects — correctness, security, contract violations — before a pipeline-generated PR is auto-merged, and to give a human operator the signal they need to make the merge call when auto-merge doesn't apply.

The pattern:

- The change was generated by one model.
- The review is performed by **other** models.
- Each reviewer sees the diff and the acceptance criteria — **not** the generator's justifications.
- Multiple reviewers vote; disagreement surfaces to the operator instead of being resolved silently.

This page documents the pattern, its guarantees, and — critically — what it does not guarantee.

## Zero-context review

Reviewers get:

- The diff.
- The acceptance criteria from the source artifact.
- The target repo's `CLAUDE.md` (project constraints, prohibited zones, style).
- The PR title and description if generated.

Reviewers do **not** get:

- The generator's chain of thought.
- The generator's rationale, explanations, or self-assessment.
- The planning-session handoff or notes that motivated the artifact.

The reason is straightforward: a review that inherits the generator's framing tends to inherit the generator's blind spots. Zero-context review forces each reviewer to independently reconstruct whether the diff actually meets the criteria as written — which is the question that matters at merge time. If the criteria are ambiguous, the reviewers will disagree, and that disagreement is the signal.

The generator's rationale isn't lost — it's in the artifact's `content` and in the digest note. It just isn't fed to the reviewers.

## Consensus semantics

Run N reviewers in parallel against the same inputs. Each returns a structured verdict:

| Field | Values | Notes |
|---|---|---|
| `verdict` | `approve` \| `changes_requested` \| `reject` | Coarse recommendation |
| `risk` | `low` \| `medium` \| `high` | Reviewer's classification of the change |
| `issues` | list of `{severity, category, summary}` | Specific findings, empty when clean |
| `notes` | free text | Reviewer commentary posted to the PR |

Recommended aggregation:

- **Unanimous `approve`** → consensus approve. Feeds the auto-merge gate.
- **Any `reject`** → consensus block. No auto-merge; operator decides.
- **Split `approve` / `changes_requested`** → surface as split verdict; operator decides. Do not silently pick a side.
- **Any reviewer flags a `high`-severity issue** → treat as block regardless of overall verdict.

N=2 is workable; N=3 gives a natural tiebreaker. Higher N buys diminishing return and more latency. Pick reviewers from different model families where possible — three variants of the same model correlate their misses.

Post every reviewer's full verdict to the PR as a comment. The operator needs to see disagreement to trust the gate.

## Risk classification as merge-policy input

Reviewer-assigned `risk` is not decoration. It is an input to the merge gate:

| Signal | Auto-merge eligible? |
|---|---|
| Repo on the auto-merge allowlist | Required |
| All reviewers `approve` | Required |
| No reviewer flags `high` severity | Required |
| Reviewer consensus `risk = low` | Required |
| CI green | Required |
| Any of the above absent | **Hold for human** |

Everything else — medium risk, changes-requested, split verdicts, CI red, unfamiliar repo — routes to the operator. The gate is designed to be conservative on the auto-merge side and permissive on the human-review side: a false negative (holding a clean PR for a human) costs a few minutes; a false positive (auto-merging a bad PR) costs credibility.

The specifics of what makes a change "low risk" are context-dependent and belong in each deployment's `CLAUDE.md` and reviewer prompt. Typical low-risk shapes: doc-only edits, additive tests, isolated bug fixes with existing coverage. Typical high-risk shapes: schema migrations, auth code, anything that touches a boundary between systems.

## Honest limits

The review loop is a **defect filter**, not a correctness proof. State this plainly to yourself and to anyone else operating the pipeline.

**What it catches.** Correctness against the stated criteria. Common security defects visible in the diff (obvious injection, hardcoded credentials, unsafe deserialization patterns). Contract violations — the change didn't touch the file it was supposed to, or touched files it was told not to. Style and convention drift.

**What it doesn't catch.** Requirements bugs — if the acceptance criteria were wrong, unanimous approval means "this correctly implements the wrong thing." Second-order effects — a change that is safe in isolation but breaks a caller in another repo. Cultural and organizational context that isn't in `CLAUDE.md`.

**The operator internalization gap.** This is the failure mode specific to autonomous pipelines: **the pipeline can build faster than the operator forms memories of what it built.** The review gate can be pristine and the PRs can all merge cleanly, and the operator can still end up unable to reason about their own system because they didn't sit with each change long enough to internalize it.

The mitigation is not "review more carefully." The mitigation is **digest notes**: for every completed artifact, write a note back to Context Library that summarizes what changed, why, and any operational implications (deployment steps, config changes, monitoring). Digest notes are for the operator's future self — the durable record that makes it possible to reason about the system in a week without paging in every merged PR.
